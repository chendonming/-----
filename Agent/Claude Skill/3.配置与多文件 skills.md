## 配置与多文件 skills

_（4 分钟）_

本视频介绍让 skills 更强大的高级技巧：完整的元数据字段集、如何编写能可靠触发的描述、在安全敏感工作流中限制工具访问权限，以及如何利用渐进式披露和多文件结构组织大型 skills。你将学会如何在支持复杂用例的同时，保持 skills 的高效性。

#### 核心要点

- **name 和 description 是必需字段** —— `allowed-tools` 和 `model` 是可选但功能强大的补充
- 优秀的描述应**回答两个问题**：这个 skill 做什么？Claude 何时应使用它？
- **allowed-tools** 可限制 skill 激活时 Claude 能使用的工具——适用于只读或安全敏感的工作流
- **渐进式披露（Progressive disclosure）**：将 `SKILL.md` 控制在 500 行以内，并链接到支持文件（参考文档、脚本、资源），Claude 仅在需要时才读取它们
- **脚本执行时不会将其内容载入上下文**——只有输出会消耗 token，从而保持上下文高效

一个基础 skill 只需 name 和 description 即可运行，但 Claude Code 中还有多种高级技巧能让你的 skills 更加高效。接下来，我们将逐一讲解关键字段、描述编写最佳实践、工具限制机制，以及如何构建大型 skills。

## Skill 元数据字段

Agent Skills 开放标准在 `SKILL.md` 的 frontmatter 中支持多个字段。其中两个是必需的，其余为可选：

- **name**（必需）—— 用于标识你的 skill。仅使用小写字母、数字和连字符，最多 64 个字符，且应与目录名一致。
- **description**（必需）—— 告诉 Claude 何时使用该 skill。最多 1,024 个字符。这是最重要的字段，因为 Claude 依靠它进行匹配。
- **allowed-tools**（可选）—— 限制 skill 激活时 Claude 可使用的工具。
- **model**（可选）—— 指定该 skill 使用的 Claude 模型。

## 编写有效的描述

指令要明确具体。如果有人告诉你“你的工作是协助处理文档”，你并不知道该做什么——Claude 也会有同样的困惑。

一个好的描述应回答以下两个问题：

- 这个 skill 做什么？
- Claude 何时应使用它？

如果你的 skill 没有在预期时机触发，请尝试在描述中加入更多与你实际提问方式匹配的关键词。Claude 正是依靠描述来判断 skill 是否相关，因此措辞至关重要。

## 使用 `allowed-tools` 限制工具

有时你希望某个 skill 只能读取文件，不能修改文件。这在安全敏感的工作流、只读任务，或任何需要设置防护边界（guardrails）的场景中非常有用。

![image](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527264%2FSkills3_17.1771527264166.png)

在此示例中，`allowed-tools` 字段被设为 `Read, Grep, Glob, Bash`。当该 skill 激活时，Claude 只能使用这些工具，且无需额外授权——无法编辑或写入文件。

```markdown
---
name: codebase-onboarding
description: 帮助新开发者理解系统的工作原理。
allowed-tools: Read, Grep, Glob, Bash
model: sonnet
---
```

如果你完全省略 `allowed-tools` 字段，该 skill 就不会施加任何限制，Claude 将使用其默认的权限模型。

## 渐进式披露（Progressive Disclosure）

Skills 与你的对话共享 Claude 的上下文窗口。当 skill 被激活时，`SKILL.md` 的内容会被载入上下文。但有时你需要引用、示例或工具脚本作为支撑材料。

将所有内容塞进一个 2,000 行的文件会带来两个问题：占用大量上下文空间，且难以维护。

渐进式披露解决了这个问题：将核心指令保留在 `SKILL.md` 中，而将详细的参考资料放入独立文件，Claude 仅在需要时才读取它们。

开放标准建议按以下方式组织 skill 目录：

- **scripts/** —— 可执行代码  
- **references/** —— 补充文档  
- **assets/** —— 图像、模板或其他数据文件  

然后在 `SKILL.md` 中通过清晰的指令链接到这些支持文件，并说明何时应加载它们：

![image](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527265%2FSkills3_13.1771527265100.png)

在此示例中，只有当用户询问系统架构时，Claude 才会读取 `architecture-guide.md`。如果用户只是问“应该在哪里添加组件”，该文件就不会被加载。这就像在上下文中放一个目录，而不是整本书。

经验法则：**将 `SKILL.md` 控制在 500 行以内**。如果超出此限制，请考虑是否应将部分内容拆分为独立的参考文件。

## 高效使用脚本

skill 目录中的脚本可以在不将其内容载入上下文的情况下执行。脚本运行后，**只有输出会消耗 token**。在 `SKILL.md` 中的关键指令是告诉 Claude **运行（run）** 脚本，而非 **读取（read）** 它。

这种机制特别适用于：

- 环境验证  
- 需要保持一致性的数据转换  
- 作为经过测试的代码比生成代码更可靠的场景
