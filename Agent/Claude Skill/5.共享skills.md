## 共享 Skills

_（4 分钟）_

当 skills 被团队或组织共享时，其价值会显著提升。本视频介绍了三种主要的分发方式——仓库提交、插件和企业级托管设置——并讲解如何配置自定义子代理使用 skills。你将了解每种方式适用的场景，并注意一个关键细节：**子代理不会自动继承 skills**。

#### 核心要点

- **项目级 skills**（位于 `.claude/skills`）通过 Git 自动共享——任何克隆仓库的人都能获得它们  
- **插件（Plugins）** 允许你通过市场将 skills 分发到多个仓库，适用于更广泛的社区使用  
- **企业级托管设置** 可在整个组织范围内部署 skills，并拥有最高优先级，非常适合强制性规范与合规要求  
- **子代理不会自动看到你的 skills**——你必须在自定义代理的 frontmatter 中显式列出 `skills` 字段  
- **内置代理**（如 Explorer、Plan、Verify）**完全无法访问 skills**——只有在 `.claude/agents` 中定义的自定义子代理才能使用  

当 skills 被共享时，其价值会成倍增长。一个仅你个人使用的 PR 审查 skill 固然有用，但若整个团队都使用它，就能统一代码审查标准，并在组织内提供一致的体验。下面我们来看看分发 skills 的不同方式。

## 通过仓库提交共享 Skills

最简单的共享方式是直接将 skills 提交到你的代码仓库中。将它们放在 `.claude/skills` 目录下，任何克隆该仓库的人都会自动获得这些 skills——无需额外安装。

当你推送更新后，团队成员在下次拉取代码时就会同步获得最新版本。这种方式非常适合：

- 团队编码规范  
- 项目专属的工作流  
- 依赖于特定代码库结构的 skills  

`.claude` 目录包含你的 agents、hooks、skills 和 settings——所有内容都通过常规 Git 工作流进行版本控制，并与团队共享。

## 通过插件分发 Skills

插件是一种跨团队和项目扩展 Claude Code 功能的方式。在你的插件项目中，创建一个 `skills` 目录，其文件结构与 `.claude` 目录类似——每个 skill 单独放在一个文件夹中，并包含一个 `SKILL.md` 文件。

将插件发布到市场后，其他用户即可自行发现并安装到他们的 Claude Code 中。

![image](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527270%2FSkills5_07.1771527270067.png)

这种方式最适合那些**不太依赖具体项目**、且对更广泛社区也有价值的 skills。

## 通过企业级托管设置部署

管理员可通过托管设置在整个组织范围内部署 skills。**企业级 skills 拥有最高优先级**——同名的个人、项目或插件 skill 都会被覆盖。

![image](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527270%2FSkills5_08.1771527270381.png)

托管设置文件还支持如 `strictKnownMarketplaces` 等功能，用于控制插件的安装来源：

```json
"strictKnownMarketplaces": [
  {
    "source": "github",
    "repo": "acme-corp/approved-plugins"
  },
  {
    "source": "npm",
    "package": "@acme-corp/compliance-plugins"
  }
]
```

这种方式适用于**必须**在全组织保持一致的强制性规范、安全要求、合规流程和编码实践。关键词就是“**必须**”。

## Skills 与子代理（Subagents）

有一个常让人意外的事实：**子代理不会自动看到你的 skills**。当你将任务委派给子代理时，它会从一个干净、全新的上下文开始。

这里有几个重要区别需要理解：

- **内置代理**（如 Explorer、Plan、Verify）**完全无法访问任何 skills**  
- **你定义的自定义子代理**可以使用 skills，但**必须显式列出**  
- skills 在子代理启动时加载，**不像主对话那样按需加载**

要创建一个带 skills 的自定义子代理，请在 `.claude/agents` 目录下添加一个代理 Markdown 文件。你也可以在 Claude Code 中使用 `/agents` 命令交互式创建：

![image](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527270%2FSkills5_13.1771527270779.png)

生成的代理文件包含一个 `skills` 字段，用于指定要加载哪些 skills。其 frontmatter 示例如下：

```markdown
---
name: frontend-security-accessibility-reviewer
description: "当你需要审查前端代码的可访问性时使用此代理..."
tools: Bash, Glob, Grep, Read, WebFetch, WebSearch, Skill...
model: sonnet
color: blue
skills: accessibility-audit, performance-check
---
```

当你将任务委派给这个子代理时，它会同时加载这两个 skills，并在每次审查中应用它们。首先确保这些 skills 已存在于你的 `.claude/skills` 目录中，然后要么创建新子代理，要么在现有代理的 Markdown 文件中添加 `skills` 字段。

这种模式在以下场景中效果极佳：

- 希望在隔离环境中委派任务，并赋予特定专业知识  
- 不同子代理需要不同的技能集（如前端审查 vs. 后端审查）  
- 希望在委派工作中强制执行标准，而不依赖提示词（prompts）
