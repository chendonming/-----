## Skills 与其他 Claude Code 功能的对比

_（3 分钟）_

Claude Code 提供了多种定制选项，选错工具可能导致不必要的复杂性。本视频将解析何时应使用 skills，何时应选用 CLAUDE.md、子代理、钩子或 MCP 服务器。你将了解每种选项的核心差异，以及它们如何在典型开发环境中协同工作。

#### 核心要点

- **CLAUDE.md** 会在每次对话中加载，最适合用于始终生效的项目级规范；**Skills** 按需加载，最适合用于任务专属的专业知识。
- **子代理（Subagents）** 在隔离的执行上下文中运行——适用于委派任务；**Skills** 则向当前对话注入知识。
- **钩子（Hooks）** 是事件驱动的（例如在文件保存或工具调用时触发）；**Skills** 是请求驱动的（根据你的提问内容激活）。
- **MCP 服务器** 提供外部工具和集成能力——属于与 skills 完全不同的类别。
- 每种功能都有其专长——**应组合使用**，而非强行将所有逻辑塞进单一机制中。

Claude Code 提供了多种定制方式：Skills、CLAUDE.md、子代理、钩子和 MCP 服务器。它们解决的是不同类型的问题。了解何时使用哪一种，能避免你构建错误的解决方案。下面我们逐一拆解。

## CLAUDE.md 与 Skills 的对比

CLAUDE.md 会在**每次对话中始终加载**。例如，如果你希望 Claude 在项目中始终使用 TypeScript 的 strict 模式，就应将其写入 CLAUDE.md 文件。

Skills 则是**按需加载**的。当 Claude 将你的请求与某个 skill 匹配成功时，该 skill 的指令才会加入当前对话上下文。例如，当你正在编写新代码时，PR 审查清单无需占用上下文；只有当你请求审查时，它才会被激活。

**使用 CLAUDE.md 的场景：**

- 始终适用的项目级规范  
- 约束条件，如“禁止修改数据库 schema”  
- 框架偏好和编码风格  

**使用 Skills 的场景：**

- 任务专属的专业知识  
- 仅在特定情境下相关的知识  
- 若放入每次对话会显得冗余的详细流程  

## Skills 与子代理（Subagents）的对比

Skills 会向**当前对话**注入知识。当 skill 被激活时，其指令会融入现有上下文。

子代理则在**独立上下文**中运行。它们接收一个任务，在隔离环境中独立执行，并返回结果，与主对话完全隔离。

**使用子代理的场景：**

- 需要将任务委派给独立执行环境  
- 需要与主对话不同的工具访问权限  
- 希望委派任务与主上下文之间保持隔离  

**使用 Skills 的场景：**

- 希望增强 Claude 在当前任务中的知识  
- 专业知识需贯穿整个对话过程  

## Skills 与钩子（Hooks）的对比

钩子是**事件驱动**的。例如，每当 Claude 保存文件时自动运行 linter，或在特定工具调用前验证输入。它们由系统事件触发。

Skills 是**请求驱动**的。它们根据你提出的问题内容决定是否激活。

**使用钩子的场景：**

- 每次文件保存时都应执行的操作  
- 特定工具调用前的输入验证  
- Claude 操作引发的自动化副作用  

**使用 Skills 的场景：**

- 指导 Claude 如何处理请求的知识  
- 影响 Claude 推理过程的准则  

## 综合运用

一个典型的配置可能包含：

- **CLAUDE.md** —— 始终生效的项目规范  
- **Skills** —— 按需加载的任务专属知识  
- **Hooks** —— 由事件触发的自动化操作  
- **Subagents** —— 用于委派任务的隔离执行环境  
- **MCP 服务器** —— 外部工具与集成  

每种机制各司其职。不要因为熟悉 skills 就把所有逻辑都塞进去——应根据场景选择最合适的工具，并可同时组合多种功能。Skills 提供自动化的任务专属知识，CLAUDE.md 用于全局指令，子代理运行于隔离上下文，钩子响应事件，MCP 则提供外部能力。

当你拥有的知识应在话题相关时自动应用，就使用 skills；并结合其他功能，实现全面而高效的定制。
