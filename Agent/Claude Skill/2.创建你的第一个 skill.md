## 创建你的第一个 skill

_（4 分钟）_

本视频将带你从头构建一个 skill —— 一个适用于所有项目的个人 PR 描述生成 skill。你将看到如何正确组织 `SKILL.md` 文件、如何测试它，并理解 Claude Code 在幕后如何发现和匹配 skills。视频还会介绍当 skill 名称冲突时，系统如何通过优先级层级决定使用哪一个。

#### 核心要点

- 一个 skill 是一个**包含 `SKILL.md` 文件的目录**，该文件在 frontmatter 中包含元数据（名称、描述），其下方是具体指令。
- Claude **仅在启动时加载 skill 的名称和描述**，然后使用语义匹配将用户请求与这些描述进行比对。
- 在 Claude 将完整 skill 内容载入上下文前，你会收到一个**确认提示**。
- 名称冲突时的优先级顺序为：**企业级 → 个人 → 项目 → 插件**
- 要更新 skill，请编辑其 `SKILL.md` 文件；要删除 skill，请删除整个目录。**每次修改后都必须重启 Claude Code** 才能生效。

接下来，我们将一步步从零创建一个 skill，然后深入探讨 Claude Code 在幕后如何加载和匹配 skills。

## 创建一个 Skill

我们将构建一个个人 skill，教会 Claude 以统一格式编写 PR 描述。由于这是个人 skill，它会存放在你的主目录中，并在所有项目中生效。

首先，在技能目录中为你的 skill 创建一个文件夹。目录名称应与 skill 名称一致：

```bash
mkdir -p ~/.claude/skills/pr-description
```

然后在该目录中创建一个 `SKILL.md` 文件。该文件由两部分组成，中间用 frontmatter 分隔线（三横线）隔开：

```markdown
---
name: pr-description
description: 编写 pull request 描述。在创建 PR、撰写 PR，或用户要求为 pull request 汇总变更内容时使用。
---

编写 PR 描述时：

1. 运行 `git diff main...HEAD` 查看当前分支上的所有变更
2. 按照以下格式撰写描述：

## What
用一句话说明这个 PR 做了什么。

## Why
简要说明为什么需要此变更。

## Changes
- 用项目符号列出具体变更
- 将相关变更归类在一起
- 提及任何被删除或重命名的文件
```

**name** 用于标识你的 skill，**description** 则告诉 Claude 何时使用它——这就是匹配依据。第二个分隔线之后的所有内容，都是 skill 被激活时 Claude 需要遵循的指令。

## 测试你的 Skill

Claude Code 在启动时加载 skills，因此创建 skill 后请重启会话。你可以通过查看可用 skills 列表来验证它是否已加载。

![image](https://everpath-course-content.s3-accelerate.amazonaws.com/instructor%2Fa46l9irobhg0f5webscixp0bs%2Fpublic%2F1771527260%2FSkills2_05.1771527260555.png)

你应该能在列表中看到你的 skill。要测试它，可以在某个分支上做些修改，然后输入类似 “为我的变更写一个 PR 描述” 的请求。Claude 会提示正在使用 PR 描述 skill，检查你的 diff，并严格按照你定义的模板生成描述——每次都保持相同格式。

## Skill 匹配机制

当 Claude Code 启动时，它会扫描四个位置查找 skills，但**只加载名称和描述**，不会加载完整内容。这一点非常重要。

当你发送请求时，Claude 会将你的消息与所有可用 skill 的描述进行比对。例如，“解释一下这个函数的作用” 可能会匹配到描述为 “用可视化图表解释代码” 的 skill，因为两者意图存在重叠。

一旦找到匹配项，Claude 会请求你确认是否加载该 skill。这个确认步骤让你清楚知道 Claude 正在引入哪些上下文。确认后，Claude 会读取完整的 `SKILL.md` 文件并执行其中的指令。

## Skill 优先级

如果你克隆的仓库中包含一个与你个人 skill 同名的 skill，系统会使用哪一个？这里有明确的优先级顺序：

- **企业级（Enterprise）** —— 由组织管理的设置，优先级最高  
- **个人（Personal）** —— 存放在主目录（`~/.claude/skills`）  
- **项目（Project）** —— 存放在仓库内的 `.claude/skills` 目录  
- **插件（Plugins）** —— 已安装的插件，优先级最低  

这种设计允许组织通过企业级 skill 强制推行标准，同时仍支持个人定制。例如，如果你公司有一个企业级的 “code-review” skill，而你自己也创建了一个同名的个人 “code-review” skill，那么企业版将优先生效。

为避免冲突，建议使用更具描述性的名称。不要只用 “review”，而应使用如 “frontend-review” 或 “backend-review” 这样的名称。

## 更新和删除 Skills

要更新 skill，请直接编辑其 `SKILL.md` 文件；要删除 skill，请删除整个目录。**任何更改后都必须重启 Claude Code** 才能生效。
